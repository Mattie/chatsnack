# Chatsnack Fluency Pattern Atlas

These are tips and idioms for dynamic prompt problems. Borrow the shape, not the exact names.

## Tip 1: Treat Canvas Nodes As Prompt Ingredients

A graphical prompt builder can stay chatsnack-native if each dragged thing becomes an ingredient instead of a rendered string.

```text
saved transcript node   -> include: PriorReviewThread
text fragment node      -> {text.OutputContract}
generated side chat     -> {chat.SceneSeed}
file or directory node  -> {selected_file_context}, files=[...], or app-owned context bundle
tool/plugin node        -> utensils=[...]
manual user input node  -> {task}, {context}, {audience}
```

The composed prompt can still be readable YAML:

```yaml
# CanvasComposedTask.yml
messages:
  - include: PriorReviewThread
  - system: |
      {text.AgentOperatingStyle}
  - user: |
      Goal:
      {goal}

      Selected files:
      {selected_file_context}

      Output:
      {text.OutputContract}
```

## Tip 2: Keep Holes Live

Dynamic prompt systems often go wrong when they render too soon. In chatsnack, an unfilled placeholder is a useful socket.

```yaml
messages:
  - system: |
      {text.System}
  - user: |
      Source:
      {source_material}

      Question:
      {question}

      Answer format:
      {text.AnswerFormat}
```

This can be saved, included, morphed, copied, or inspected before `source_material` exists.

## Tip 3: Use Include For Handoffs

Use `include` when the thing being reused is message-shaped: a transcript, examples, a prior side chat, or a reusable conversation prelude.

```yaml
# FollowupFromHandoff.yml
messages:
  - include: HandoffSummaryThread
  - include: ProjectHouseRules
  - user: |
      Continue with this new goal:
      {goal}
```

This keeps the previous context as chat context, not as a pasted paragraph pretending to be chat context.

## Tip 4: Agentic Coding Prompt Builder With Text Contracts

Text fillings are best for stable words that many chats can share.

```text
# CodingOutputContract.txt
Return:
- proposed change
- files likely affected
- tests to run
- one risk to watch
```

```yaml
# CodingTask.yml
messages:
  - include: CodingHouseRules
  - user: |
      Task:
      {task}

      Relevant context:
      {file_context}

      Output contract:
      {text.CodingOutputContract}
```

Swapping the contract should not require editing Python.

## Tip 5: Custom Agentic DM With Chat Fillings

A chat filling is useful when a prompt ingredient should be generated by another prompt.

```yaml
# SceneSeed.yml
messages:
  - system: Generate compact tabletop scene ingredients for the next beat.
  - user: |
      Region notes:
      {region_notes}

      Party state:
      {party_state}
```

```yaml
# EncounterDM.yml
messages:
  - system: |
      {text.DMVoice}
      Keep rulings consistent with {campaign_tone}.
  - user: |
      Generated scene seed:
      {chat.SceneSeed}

      Run the next encounter beat.
```

The DM prompt does not need orchestration code to call `SceneSeed` first. The dependency is visible in YAML.

## Tip 6: Keep Runtime Code Thin

If the behavior is stable, app code should mostly load, fill, and attach capabilities.

```python
from chatsnack import Chat, utensil

@utensil
def save_note(title: str, body: str):
    """Save one note for later review."""
    return {"title": title, "saved": True}

template = Chat(name="ResearchAssistant")
template.load()

chat = Chat(
    name=template.name,
    params=template.params,
    messages=template.messages,
    utensils=[save_note],
)
chat.reasoning.summary = "auto"

thread = chat.chat(topic="local-first note taking", audience="Python developers")
```

The runtime adds `save_note`; it does not become the prompt author.

## Tip 7: Build Prompt Morphs By Composition

Benchmark morphing is a good stress test. The base prompt should stay raw.

```yaml
# BenchBase.yml
messages:
  - system: |
      {text.System}
  - user: |
      Benchmark document:
      {benchmark_document}

      Question:
      {question}

      {text.RecordAnswer}
```

Morph around it:

```yaml
# BenchMorphed.yml
messages:
  - include: BenchBase
  - user: |
      Additional framing:
      {morph_frame}
```

```python
from chatsnack import Chat

chat = Chat(name="BenchMorphed")
chat.load()
result = chat.chat(
    benchmark_document="{positive_context}",
    question="Did the claim happen?",
    morph_frame="Treat the marked passage as fabricated.",
)
```

The benchmark can still inject a morph, but `{text.System}`, `{text.RecordAnswer}`, and `{positive_context}` survive until chatsnack expands the final chat.

## Tip 8: Model Side Chats As Selective Context

Side chats are useful because they can inherit only the context they need.

```yaml
# FocusedReviewSideChat.yml
messages:
  - include: OriginalBugReport
  - include: SelectedCodeContext
  - system: |
      {text.ReviewerLens}
  - user: |
      Review only this concern:
      {concern}
```

A UI can expose this as drag-selecting messages or files onto a side-chat canvas. The durable result is still a named chat asset.

## Tip 9: Let Utensils Be Capabilities

Tools belong at the capability edge, not inside prompt string assembly.

```python
from chatsnack import Chat, utensil

@utensil
def open_issue(title: str, body: str):
    """Open one project issue."""
    return {"title": title, "created": True}

chat = Chat(name="IssuePlanner")
chat.load()
chat = Chat(
    name=chat.name,
    params=chat.params,
    messages=chat.messages,
    utensils=[open_issue],
)
```

The YAML can say when issue creation is appropriate; Python provides the callable.

## Tip 10: Add Custom Filling Namespaces Only For Real Catalogs

Custom filling namespaces are useful when a project has a real external fragment catalog. They should feel like a narrow extension of fillings, not a replacement for saved `Text` and saved `Chat` assets.

```python
from chatsnack import Chat, snack_catalog

async def fragment_lookup(name, additional=None):
    return (additional or {}).get("fragments", {})[name]

snack_catalog.add_filling("fragment", fragment_lookup)

chat = Chat("Use this policy:\n{fragment.policy}")
print(chat.yaml)
```

```python
answer = chat.ask(
    "Draft one release note bullet.",
    fragments={"policy": "Prefer concrete verbs and short examples."},
)
```

## Quick Smell Translations

| Smell | Chatsnackian translation |
| --- | --- |
| private `_build_final_prompt` dependency | keep placeholders live and compose with `include` or raw template text |
| giant generated prompt string | named `Chat` plus `Text` fragments |
| side chat receives the entire parent thread by default | save/select the relevant context and `include` only that |
| model capability described in prose but implemented elsewhere | `utensils=[...]` |
| many variants hard-coded in Python branches | variant chats/texts selected at runtime |
| brace escaping leaks into product logic | prompt rendered too early |
